# ProjectLens 類似課題検索(ローカルRAG) 要件定義書(v0.4)

> **リリース**: v0.4(ローカル埋め込みによる類似課題検索 + 過去事例の解決策要約)
> **ステータス**: 確定(2026-06-13、`/refine-requirements v0.4` の壁打ちにより確定)
> **前提**: v0.3(FoundationModels 推論基盤 + `job_queue` / `ai_results` + AI 表示 UI)の完了
> 元ドラフト「AIユースケース拡充 / MLX バックエンド」からの再分割: ユースケース群のうち **検索基盤(RAG)** を v0.4 に確定。生成ダイジェスト群(横断サマリ・週次/月次レポート・背景/決定事項要約)は `../v0.4.5/requirements.md` へ、MLX バックエンドはさらに後続へ分離した。

## 背景と目的

v0.3 で実装した **FoundationModels** を、検証機(macOS 26.3.1 + Apple Intelligence 有効、Apple Silicon)で実際に sidecar 直接起動して確認したところ、`availability: true` で日本語の要約・リスク判定・対応提案が**実用品質**で生成された。これにより:

- MLX バックエンドの2大目的のうち「品質向上の選択肢」は緊急度が下がった(3B級でも実用)。「古い macOS のカバー」は他ユーザー向けの価値であり、全面無料開放中の現段階では優先度を下げ、**後続リリースへさらに分離**する(`LlmInference` trait 抽象は v0.3 で準備済み)。

v0.4 は ProjectLens 最大の差別化「**複数プロジェクト横断 × ローカル完結**」を、ユーザー価値の高い「**類似課題・過去事例の検索(ボタンひとつ)+ 解決策要約**」として実装する。

### 設計上の確定事項(壁打ちより)

- **コメントを埋め込み対象に含める**: 解決策・決定事項はコメントに書かれることが多く、タイトル+本文だけの埋め込みでは「過去事例の解決策」を取りこぼす。よってコメント取得を v0.4 に取り込み、埋め込み対象は「タイトル + 本文 + コメント」とする。
- **コメントは差分取得**: 全課題に毎同期で取得せず、変更課題のみ・新規コメントのみを取得して API レート負荷を抑える(Backlog `GET /issues/:id/comments` の `minId` を利用)。
- **完了課題を期間指定でコーパスに取り込む**: 解決ノウハウが最も詰まっているのは完了(Closed)課題だが、既存同期は `statusId=[1,2,3]`(未対応・処理中・処理済み)のみで **完了(status 4)を除外**している。検索コーパスには完了課題を**設定で指定した期間ぶん**取り込む(`statusId=4` + `updatedSince`)。ダッシュボード等の既存表示は従来どおり完了を除外し、**検索コーパスと UI 表示を分離**する。
- **ローカル完結を維持 / 既定は OS 組み込み埋め込み**: Apple OS 同梱の文埋め込み(`NLEmbedding.sentenceEmbedding`)は対応7言語に**日本語を含まない**ため使えない。FoundationModels も生成専用で埋め込み API を公開していない。一方 **`NLContextualEmbedding`(NaturalLanguage・macOS 14+)は日本語/CJK 対応**で、検証機での実測(`dimension=512`・`hasAvailableAssets=true`・類似文 0.81〜0.85 > 無関係 0.65〜0.71)から **top-N の「関連課題提示」には実用**と判断。よって **v0.4 既定の埋め込みは `NLContextualEmbedding`(512次元・アプリ同梱不要・即動作)** とし、トークン文脈ベクトルを mean-pooling して文ベクトルを得る。検索精度をさらに上げたいユーザー向けに、より高精度なモデルを HuggingFace から DL できる仕組みは**将来の有料機能候補**(下記スコープ外)とし、`EmbeddingBackend` trait で差し替え可能にしておく。

## アーキテクチャ方針(確定)

v0.3 の「バックグラウンド処理 → SQLite 保存 → UI は読むだけ」を踏襲し、埋め込みも sync 連動のバックグラウンドで構築する。

```
Backlog API sync(既存: statusId=[1,2,3])
   + 完了課題の期間取り込み(新規: statusId=4 + updatedSince=設定期間)
    ↓ 新規・更新課題を検出
変更課題のコメントを差分取得(minId)
    ↓
SQLite (issues / comments)
    ↓ (title + body + comments) を前処理 → job_queue(embed)
バックグラウンドワーカー(オンデバイス埋め込み生成: NLContextualEmbedding / sidecar 経由)
    ↓
SQLite (issue_embeddings: 512次元 BLOB)
    ↓
課題起点「類似を探す」→ 対象ベクトル × 全課題ベクトルのコサイン類似度(Rust 総当たり)
    ↓ 上位N件
FoundationModels で過去事例の解決策を要約(v0.3 の推論基盤を再利用)
    ↓
Vue UI(類似一覧 + 解決策要約)
```

## 機能要件

### FR-V04-001: ローカル埋め込み生成基盤

- **優先度**: 必須
- **説明**: OS 組み込みの埋め込みモデルでオンデバイスにベクトルを生成する(既定はアプリ同梱不要)。外部送信なし。
- **受け入れ基準**:
  - 既定バックエンドは **OS 組み込み `NLContextualEmbedding(language: .japanese)`(512次元)**。トークン文脈ベクトルを mean-pooling して文ベクトルを得る。OS がアセットを提供するため**アプリ同梱不要**(多くの macOS 26 環境で `hasAvailableAssets == true`)。
  - 日本語中心コーパスに合わせ言語は `.japanese`(CJK)固定とし、全ベクトルを同一空間に保って比較可能にする(英語混在文も同モデルで埋め込む)。
  - 埋め込み生成はオンデバイスで完結し、課題データ・コメントを外部へ送信しない(NFR-V04-001)。
  - 既存 Swift sidecar に embed リクエストを追加し、Rust から `EmbeddingBackend` trait 経由で呼べるようにする。`LlmInference`(生成)とは別経路。保存するモデル名は実際にベクトルを生成したバックエンドの `model_name()` を単一の真実源とする(再埋め込み判定の正確性)。
  - プレフィックス(`query:` / `passage:`)はワイヤ契約として保持するが、`NLContextualEmbedding` は使用しない(将来の e5 系バックエンド用に sidecar 側で解釈)。
  - アセット未取得・利用不可(Intel 機等)の環境では**検索機能のみ**を無効化し、既存機能・v0.3 の AI 機能を一切阻害しない。

### FR-V04-002: コメントの差分取得

- **優先度**: 必須
- **説明**: 検索品質向上(および v0.4.5 の背景要約)のためコメントを取得する。レート負荷を抑える差分取得とする。
- **受け入れ基準**:
  - `GET /api/v2/issues/:issueIdOrKey/comments` を `minId`(課題ごとに保存した最終取得コメントID)・`order=asc`・`count=100` で呼び、**新規コメントのみ**取得する。
  - 取得対象は v0.3 の差分検出で「新規・更新」と判定された課題のみ。初回(埋め込み未構築時)はコーパス対象の全課題に対し1回だけ全件取得する。
  - 課題ごとの最終取得コメントID・取得状態を SQLite に保存する(例: `issue_comment_state`)。コメント本文は SQLite に保存する(例: `issue_comments`)。
  - レートリミット(`rate_limit`)を尊重し、残量が少ない場合はバックオフして次サイクルへ繰り越す。sync・UI をブロックしない。
  - コメント API は Backlog API v2 準拠。取得失敗時はスキップして記録し、リトライ上限を設ける。

### FR-V04-003: 完了課題の期間指定コーパス取り込み(設定可変)

- **優先度**: 必須
- **説明**: 完了(Closed)課題を検索コーパスにのみ取り込む。取り込み期間をユーザーが設定で指定できる。
- **受け入れ基準**:
  - 設定画面に「過去完了課題の取り込み期間」を追加する。範囲は **1〜12ヶ月、最大2年(24ヶ月)** で可変。既定値は実装時に決定(目安 3〜6ヶ月)。
  - 同期時、設定期間に基づき `statusId[]=4` + `updatedSince=<現在 - 期間>` でページング取得(`count=100` + `offset`)し、完了課題を取り込む。
  - 取り込んだ完了課題は**検索コーパス専用**として扱い、ダッシュボード・課題一覧・スコアリング・通知などの既存表示には**従来どおり含めない**(完了除外を維持)。
  - 期間を短縮した場合、範囲外になった完了課題とその埋め込み・コメントをクリーンアップする。
  - 取り込み件数・進捗を設定画面で確認できる(初回・期間変更時は件数が増えるため)。

### FR-V04-004: ベクトル保存と類似検索

- **優先度**: 必須
- **説明**: 埋め込みを保存し、複数プロジェクト横断で課題間の類似度を計算する。
- **受け入れ基準**:
  - SQLite に `issue_embeddings`(`issue_id`, `model`, `dim`, `vector`(BLOB), `source_hash`, `updated_at`)を追加する。
  - 埋め込み対象テキスト = 課題タイトル + 本文(切り詰め) + コメント本文(前処理・上限つき)。512トークン上限を考慮した前処理を行う(チャンク分割で複数ベクトル/課題、または課題ダイジェスト生成→単一ベクトル。方式は未解決事項)。
  - 検索は対象課題ベクトルと全課題ベクトルのコサイン類似度を Rust で計算し、上位N件を返す(数千件規模で UI 応答性を維持)。将来 `hnsw_rs` / `sqlite-vec` へ移行できる構造にする。
  - 検索は**同一ワークスペース内で複数プロジェクト横断**で行う(差別化)。クエリ課題自身は結果から除外する。
  - `source_hash` により入力テキストが不変なら再埋め込みしない(無駄な再計算を避ける)。

### FR-V04-005: 課題起点の類似検索 UI + 解決策要約

- **優先度**: 必須
- **説明**: 「ボタンひとつ」で類似・過去事例を提示し、過去の解決策を要約する。
- **受け入れ基準**:
  - 課題カード(`IssueCard`)および課題詳細ダイアログ(`IssueDetailDialog`)に「**類似を探す**」ボタンを追加する。
  - 押下で、選択課題をクエリに**横断**で類似上位N件を一覧表示する(プロジェクトキー・課題キー・タイトル・類似度・ステータス・担当者)。完了課題も結果に含める。
  - 上位群について FoundationModels(v0.3 基盤)で「**過去の対応・解決策の要点**」を要約表示する。完了課題があれば解決の要点を抽出する。すべての AI 生成表示に生成ラベル(`mdi-creation` 等)を付与する。
  - 各候補からブラウザで開く動線(既存)を維持する。埋め込み未構築の課題では理由(構築待ち)を提示する。
  - 出力言語は UI 言語(ja/en)に追従する。

### FR-V04-006: リスク判定への期限超過(遅延日数)の織り込み

- **優先度**: 必須(v0.3 からの持ち越し。2026-06-13 にユーザーと合意のうえ v0.4 へ deferred した項目)
- **説明**: v0.3 のリスク判定は**課題の内容だけ**で決まり、期限超過の度合いが反映されない(実例: `NCA_PM-12` は **469日超過でも risk=medium**)。最終リスクを `max(内容リスク[LLM], スケジュールリスク[遅延日数しきい値])` で決定的に底上げする。日付計算は SQL のまま決定的に保ち、リスクを LLM 依存にしない。
- **受け入れ基準**:
  - `worker.rs::process_job` で `delay_days` 算出後、`final_risk = max(llm_risk, schedule_risk(delay_days))` を保存する。`schedule_risk` は決定的ヘルパーとして実装する。
  - しきい値(要調整): 14日超 = high / 1〜13日超 = medium 以上 / 期限間近(数日以内)= medium / それ以外は内容リスクのまま。`delay_days` は 正=超過・負=猶予・0=当日。
  - 既存の `ai_results` は **LLM 再実行なし**で再計算できる(保存済みの内容リスクと `delay_days` から max を取るだけ)。再計算の適用手段(マイグレーション or 起動時バッチ)を用意する。
  - リスクが期限超過により昇格した場合、その理由を UI で説明可能にする(任意)。
  - 代替案「遅延日数をプロンプトに渡して LLM に総合判断させる」はリスクが LLM 依存で不確実なため非推奨(採用するなら本決定的方式と併用)。

## 非機能要件

### NFR-V04-001: プライバシー(ローカル完結)

- 埋め込み生成・類似検索・解決策要約はすべてオンデバイスで完結する。課題データ・コメントを外部(クラウド API 等)へ送信しない。v0.3 の差別化(ローカル完結・外部送信なし)を維持する。

### NFR-V04-002: 性能

- 数千件規模で類似検索の UI 応答は 100ms 目安(初回のベクトル計算・モデルロードを除く)。
- 埋め込み生成・コメント取得はバックグラウンドで実行し、同時実行を制御する。sync・UI 操作をブロックしない。

### NFR-V04-003: リソース消費

- 埋め込みモデルのメモリ常駐を抑制する(アイドル時=キュー空で CPU を消費しない、v0.3 と同方針)。
- 初回の全件埋め込み・完了課題取り込みはバックグラウンドで段階的に実行し、UI/sync をブロックしない。

### NFR-V04-004: 互換性・配布

- 既定の `NLContextualEmbedding` は OS 提供のため**アプリ配布サイズを増やさない**(同梱モデル不要)。将来 HuggingFace から DL する高精度モデルはユーザー任意・DL 基盤側で管理する。
- 埋め込み実行は Apple Silicon / macOS 14+(`NLContextualEmbedding`)を前提とする。Intel 機・アセット未取得環境では検索のみ無効化する。

### NFR-V04-005: 堅牢性

- 埋め込みモデル・コメント API の失敗時もアプリ本体・既存機能・v0.3 AI は継続動作する。検索機能のみ degrade させ、理由を表示する。

## スコープ外(やらないこと)

- **生成ダイジェスト群**(複数プロジェクト横断サマリ・週次/月次レポート・課題の背景/決定事項要約)→ **v0.4.5**(コメントデータは v0.4 で収集済みのものを再利用)
- **任意キーワード/自由文の検索専用画面** → 将来(v0.4 は**課題起点**の「類似を探す」のみ)
- **MLX(Qwen3)バックエンド**(3ティアのモデル DL・ティア選択 UI・バックエンド選択ロジック)→ 後続リリース(v0.3 で 3B 級が実用品質と確認できたため優先度を下げた。`LlmInference` trait 抽象は準備済み)
- **課金・ライセンス・Free/Pro ゲート** → v0.5
- **高精度埋め込みモデルの HuggingFace ダウンロード**(アプリ指定モデルを DL して既定の OS 組み込み埋め込みと差し替え)→ **将来・有料機能候補**。DL 基盤(進捗/中断再開/検証/削除)は分離した MLX バックエンドのダウンロード基盤と共通化できる。`EmbeddingBackend` trait で差し替え可能な前提は v0.4 で用意済み
- **ベクトル DB 拡張(sqlite-vec)/ ANN(hnsw)** → スケール時に移行(v0.4 は総当たりコサイン)
- Backlog API への書き込み、Windows(Candle)対応、Gemma 4 対応 → 将来

## 未解決事項(実装中に判断)

- **埋め込み戦略**: コメント込みテキストの 512トークン対策。現状は「タイトル+本文+コメントを結合し文字数上限で切り詰めた単一テキスト → 単一ベクトル」を既定採用。チャンク分割で複数ベクトル/課題にするか、課題ダイジェストを生成して単一ベクトルにするかは継続検討(後者は v0.4.5 の背景要約と生成物を共有できる)。
- **NLContextualEmbedding の品質限界**: mean-pooling した文ベクトルは表層テーマで誤判定する場合がある(実測で「同期失敗」と「マイグレーション失敗で落ちる」を混同。最近傍 4/6)。top-N 提示では実用だが、精度要求が高い場合は HuggingFace DL モデル(有料機能候補)への差し替えを検討。
- **言語の扱い**: `.japanese` 固定のため英語中心チケットの埋め込みは準最適。多言語混在ワークスペースでの品質は実データで確認。
- **完了課題取り込み期間の既定値**と、初回・期間拡大時の全件取得のレート制御(大規模ワークスペースでの所要時間・分割実行)。
- **類似検索のパラメータ**: 上位N・類似度しきい値(暫定 0.80)・完了/未完了の重み付け。
- **再埋め込みポリシー**: モデル更新時(OS 組み込み → DL モデル等で次元も変わる)・本文/コメント変更時の再生成。`source_hash` + `model` で判定。
- **sqlx からの拡張ロード可否**(将来 `sqlite-vec` 採用時の結線実績は未確認)。
