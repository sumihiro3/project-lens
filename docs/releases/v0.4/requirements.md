# ProjectLens AIユースケース拡充 + MLX バックエンド 要件定義書(v0.4 ドラフト)

> **リリース**: v0.4(AIユースケース拡充 / MLX バックエンド追加)
> **ステータス**: ドラフト(着手時に `/refine-requirements v0.4` で確定させること)
> 元ドラフト: AI機能 追加要件定義書(2026-06-12)の「3. AI 機能ユースケース」と「1. LLM 推論バックエンド(MLX)」「2. モデル管理」より分離。
> 前提: v0.3(FoundationModels 推論基盤 + チケット単位ユースケース)の完了

## 目的

v0.3 で確立した基盤(LlmInference / job_queue / ai_results)の上に、(a) ユースケースの拡充、(b) MLX(Qwen3)バックエンドの追加、の2軸を進める。両方を v0.4 に含めるかは確定時に判断する(v0.3 の品質フィードバック次第で優先度が変わる)。

## (a) MLX(Qwen3)バックエンド追加

v0.3 の壁打ちで「v0.3=OS搭載モデルのみ、MLX は後続」と確定した分。目的は2つ:

- **古い macOS のカバー**: FoundationModels は macOS 26+ 限定。macOS 25 以前・Apple Intelligence 無効環境に AI 機能を届ける
- **品質向上の選択肢**: 約3B級の OS モデルで品質が不足するケースに、より大きいモデルを提供する

確定済みの設計方針(v0.3 壁打ちより):

- Swift sidecar (MLX) で実装、`LlmInference` のバックエンドとしてレジストリ追加
- モデルは Qwen3 系3ティア(4bit): 軽量=4B(約2.5GB、8GB機) / 推奨=8B(約5GB、16GB機) / 高精度=14B(約8GB、24GB機以上)
- HuggingFace(mlx-community)からダウンロード。進捗 emit・中断再開・検証・削除を実装
- Gemma 4(E4B/12B)は MLX Swift の `gemma4` アーキテクチャ対応後にレジストリ追加で検討

## (b) ユースケース拡充

**複数プロジェクト横断**(ProjectLens 最大の優位点)を中心に、Backlog AIアシスタント公式プロンプト集(25種)のうちローカル実現可能な残りを積む。

## 候補ユースケース(元ドラフトの優先度分類)

### 優先度: 高(プロジェクト単位・1日1回まとめて生成)

- 複数プロジェクト横断の状況把握 / 分析
- マイルストーン別の進捗整理・要約
- 更新が一定期間止まっている課題の抽出(SQL一発、軽量)
- 月次 / 週次の活動レポート・サマリー生成
- 新規参加者向けオンボーディング資料生成
- 課題の背景・決定事項の要約(コメント取得が必要 → Backlog API 拡張要検討)

### 優先度: 中(入力データに依存)

- 担当者の自己評価資料の整理(期間指定 + LLM)
- 類似課題・過去エラー事例の検索 + 解決策要約(キーワード検索 or ベクトル検索)
- 海外メンバー向けの英語要約・翻訳
- 課題テンプレート / マニュアルの生成
- 週ごとの作業時間集計(Backlog に工数データがある場合のみ)

### 部分的・精度限定(期待値を下げて提供)

- メンバーのスキル・対応傾向の推論(集計は可能、性格傾向の推論は限定的)
- 業務内容に対する担当候補者の提案

## 確定時に詰めること

- (a) MLX バックエンドと (b) ユースケース拡充を同一リリースにするか分割するか(v0.3 の品質フィードバックで判断)
- v0.4 に含めるユースケースの取捨選択と優先順位
- プロジェクト単位処理のスケジュール(1日1回の実行タイミング)と UI(レポート画面の新設?)
- コメント・更新履歴の取得(Backlog API 呼び出し増)とレートリミットの兼ね合い
- 複数バックエンド共存時の既定選択ロジック(環境検出で自動 or ユーザー選択)
- モデル品質が不足する場合の評価ハーネス再検討(v0.3 スコープ外とした件)
