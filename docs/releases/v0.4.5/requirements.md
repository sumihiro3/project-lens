# ProjectLens レポート/サマリー(生成ダイジェスト) 要件定義書(v0.4.5)

> **リリース**: v0.4.5(複数プロジェクト横断サマリ + 週次/月次レポート + 課題の背景/決定事項要約)
> **ステータス**: 確定(2026-06-13、`/refine-requirements v0.4.5` の壁打ちにより確定)
> **前提**: v0.4(ローカル RAG 検索基盤 + コメント差分取得 + `ai_results` per-issue 要約/リスク)の完了。**新規 API 呼び出し・新規 per-issue LLM 生成は原則不要**で、v0.4 までに収集・生成済みのデータ(`issues` / `issue_comments` / `ai_results`)を再利用する。
> 元ドラフト: 旧 v0.4「AIユースケース拡充」のうち**生成ダイジェスト系**を分離したもの(検索基盤=RAG は v0.4 で確定)。

## 背景と目的

v0.3 の **FoundationModels** 生成基盤と、v0.4 で収集した**コメントデータ**・生成済みの **`ai_results`(課題ごとの1行要約・リスク・遅延日数)** の上に、**専用「レポート/サマリー」画面**を新設し、プロジェクト単位・横断の生成ダイジェストを提供する。

ProjectLens 最大の差別化は「**複数プロジェクト横断 × ローカル完結**」である。個々の課題を見るだけでは得られない「**今、どのプロジェクトの何に注意すべきか**」という横断の俯瞰を、外部送信なしのオンデバイス生成で提供する。1日1回のバックグラウンド生成を基本とし、UI を開いたときには結果が準備済みの状態にする。

### 設計上の確定事項(壁打ちより)

1. **スコープは3機能すべて**: ①複数プロジェクト横断サマリ ②週次/月次アクティビティレポート ③課題の背景・決定事項要約。①②は新「レポート/サマリー」画面に集約し集約基盤を共有、③は課題詳細(`IssueDetailDialog`)での per-issue 要約として別 UI 面で提供する。

2. **生成方式は「既存 AI 結果 + SQL 集約の再利用」**(コンテキスト上限対策の確定): FoundationModels(3B 級)は guided generation スキーマと合算でコンテキストが厳しく、多数課題を直接投入できない。よって①②は **map-reduce(課題ごと再要約)を採らず**、(a) **数値は SQL で決定的に集計**(件数・期限超過・停滞・新規/完了数)し、(b) **`ai_results` の per-issue 1行要約・リスク**から注目上位 N 件のみを LLM に渡して**短い narrative(注目点)だけを生成**する。新規 per-issue LLM 呼び出しはゼロ。v0.4 の `summarize_solutions` と同じ「既存生成物の再利用」思想。**数値は決定的・文章は LLM** という責務分離により、コンテキストを最小化しつつ数値の正確性を担保する。

3. **生成経路は既存 `analyze` 経路の流用(Swift sidecar 改修なし)**: v0.4 の `summarize_solutions` と同様、集約済み context を `AiAnalysisInput.description_head` に載せて `infer` を呼び、構造化出力(`summary` = 見出し1行 / `suggestion` = 注目点の本文)を narrative にマップする。`risk_level` は集計済み数値で上書きするため LLM 出力を信用しない。sidecar の再ビルド・codesign・再配布を要する自由文 `summarize` 経路の新設は**見送り**(将来、レポート品質向上が必要になれば移行可能。`summarize_solutions` も同時に移行できる)。

4. **保存は「履歴あり・エクスポートなし」**: 週次/月次レポートは**期間キー**(`2026-W24` / `2026-06`)で履歴を残し過去を閲覧できる。横断サマリは**最新を上書き**(`latest`)。Markdown 等での外部エクスポート/共有は将来。

5. **出力言語は UI 言語追従のみ(ja/en)**: v0.4 と同方針。生成時の UI 言語で生成・保存し、その言語で表示する。日英同時生成(海外メンバー向け英語版併存)は将来。

## アーキテクチャ方針(確定)

v0.3/v0.4 の「バックグラウンド生成 → SQLite 保存 → UI は読むだけ」を踏襲する。①②(集約レポート)は**低頻度・1ワークスペースあたり数件**のため、v0.4 の `summarize_solutions` と同じく **`job_queue` を介さずコマンド/スケジューラのバックグラウンドタスクから直接 `create_backend` → `infer` を呼ぶ**(sidecar の直列管理タスクで analyze/embed と自然に直列化される)。③(per-issue 背景要約)は**オンデバンド生成 + キャッシュ**とする。

```
[① 横断サマリ / ② 週次・月次レポート] — 低頻度・WS単位の集約
  scheduler(1日1回判定: 前回生成からの経過 / 期間ロールオーバ) ── または 手動「再生成」
      ↓
  SQL 集約(決定的): プロジェクト別の 未完了/期限超過/停滞/新規/完了 件数・リスク分布
      ↓ + ai_results の注目上位N件(1行要約・リスク)だけを抽出
  compact context(<3000字)を組み立て → create_backend(FoundationModels)→ infer(analyze 流用)
      ↓
  SQLite (report_summaries: stats_json[決定的] + headline + narrative + lang + period_key)
      ↓
  Vue UI(/reports 画面: 統計テーブル[決定的] + AI narrative[生成ラベル付き])

[③ 課題の背景・決定事項要約] — per-issue・オンデマンド
  IssueDetailDialog で「背景・経緯を要約」 ── コメント(issue_comments)を context 化
      ↓ source_hash 一致ならキャッシュ即返し(再生成しない)
  create_backend → infer(analyze 流用)→ SQLite(issue_background_summary: cache)
      ↓
  IssueDetailDialog に要約表示(生成ラベル付き)
```

## 機能要件

### FR-V045-001: 専用「レポート/サマリー」画面の新設

- **優先度**: 必須
- **説明**: ダッシュボードを肥大化させず、横断サマリ・週次/月次レポートを集約する専用画面を新設する。
- **受け入れ基準**:
  - `src/pages/reports.vue` を新設し、`src/app.vue` の `v-navigation-drawer` に「レポート/サマリー」項目(`to="/reports"`、アイコン `mdi-file-chart` 等)を追加する。i18n キー `app.reports` を日英に追加する。
  - 画面は「横断サマリ」セクションと「週次/月次レポート」セクションを持つ。複数ワークスペース有効時はワークスペース選択(または有効ワークスペースごとに並べる)を提供する(横断は**同一ワークスペース内のプロジェクト横断**。別ワークスペース=別 Backlog スペースは混在させない)。
  - すべての AI 生成テキストに生成ラベル(`mdi-creation` + `ai.settings.generated`)を付与する(v0.3/v0.4 の `ai-text-box` スタイルを踏襲)。
  - ページは 50-100 行目安・ロジックは composable(`useReports`)へ分離する(`docs/COMPONENT_RULES.md`)。
  - AI 非対応環境・データ未生成時はセクションを degrade 表示し(理由提示)、既存機能・v0.3/v0.4 機能を一切阻害しない。

### FR-V045-002: 複数プロジェクト横断サマリ(既存 AI 結果 + SQL 集約の再利用)

- **優先度**: 必須
- **説明**: 有効な全プロジェクトを横断した状況把握を 1日1回生成し、「**今、注意すべきこと**」を横断で提示する(ProjectLens 最大の差別化)。
- **受け入れ基準**:
  - **数値は SQL で決定的に集計**する(LLM に数えさせない): ワークスペース内のプロジェクト別に「未完了件数・期限超過件数・停滞件数(一定日数更新なし)・自分担当の要対応件数・リスク分布(`ai_results.risk_level` の high/medium/low)」を算出する。停滞しきい値・対象期間は定数で一元定義する。
  - **narrative は注目上位 N 件のみから生成**する: 期限超過・高リスク・停滞などで重み付けした上位 N 件の課題について、`ai_results` の**生成済み1行要約**(`ai_summary`)・リスク・プロジェクト・遅延日数だけを compact context(`< CONTEXT_MAX_CHARS`(≈3000字))に詰め、`analyze` 流用で「注目点(数文)」と「見出し1行」を生成する。新規 per-issue LLM 呼び出しは発生させない。
  - 横断サマリは**最新を上書き保存**する(`report_summaries` の `report_type='cross_summary'` / `period_key='latest'`)。`stats_json`(決定的集計)・`headline`・`narrative`・`lang`・`generated_at` を保持する。
  - UI は決定的な統計テーブル(プロジェクト別カウント。クリックで該当課題一覧へ絞り込み遷移できれば望ましい・任意)と、AI narrative(生成ラベル付き)を表示する。
  - `ai_results` が1件も無い/AI 非対応の場合は、統計テーブルのみ表示し narrative は「未生成(AI を有効化してください 等)」へ degrade する(NFR-V045-003)。

### FR-V045-003: 週次/月次アクティビティレポート(期間キーで履歴保持)

- **優先度**: 必須
- **説明**: 期間内の更新・完了・新規課題を集約してレポート生成する(プロジェクト単位/横断)。期間ごとに履歴を残す。
- **受け入れ基準**:
  - 期間は**週次(ISO 週: `YYYY-Www`)**と**月次(`YYYY-MM`)**の2種。各期間の「新規作成・更新・完了(コーパス取り込み済みの完了課題を活用)」件数をプロジェクト別に SQL 集計する(決定的)。
  - 完了課題の活用: v0.4 で取り込んだ `is_corpus_only` の完了課題(`statusId=4`)から、期間内に完了(更新)した課題を集計・列挙する。コーパス取り込み期間(`corpus_months`)外は対象外である旨を UI で補足する。
  - narrative は FR-V045-002 と同じ方式(注目上位 N 件 + 既存要約の再利用 + `analyze` 流用)で「期間のハイライト(主な完了・新たな期限超過・リスク増)」を生成する。
  - 期間キーで履歴保存する(`report_summaries` の `report_type='weekly'|'monthly'` / `period_key`=期間キー)。同一期間の再生成は UPSERT で上書きし、過去期間は残す。UI は期間セレクタで過去レポートを閲覧できる。
  - 生成タイミングは FR-V045-005 に従う(週/月のロールオーバ検知 + 手動再生成)。

### FR-V045-004: 課題の背景・決定事項の要約(per-issue・コメント再利用)

- **優先度**: 必須
- **説明**: コメントを読んで課題の経緯・決定事項を要約する。v0.4 で取得済みのコメント(`issue_comments`)を利用し、**新規 API 呼び出しは不要**。課題詳細での per-issue 提供。
- **受け入れ基準**:
  - `IssueDetailDialog` に「**背景・経緯を要約**」導線(ボタンまたは折りたたみセクション)を追加する。`useSimilarSearch`(類似を探す)・`reanalyze`(再分析)と並ぶ AI アクションとして配置する。
  - 押下で当該課題のコメント本文(`get_comments_text`)を context 化し、`analyze` 流用で「経緯・決定事項の要点(数文)」を生成・表示する。出力言語は UI 言語追従。
  - **キャッシュ**: コメントの `source_hash`(v0.4 の埋め込みと同じ SipHash 方式)と `lang` をキーに `issue_background_summary`(`workspace_id`, `issue_id`, `lang`, `summary_text`, `source_hash`, `generated_at`)へ保存し、コメント不変かつ同一言語なら**再生成せずキャッシュを即返す**(LLM・sidecar を起こさない・NFR-V045-002)。
  - コメントが無い課題は生成せず「コメントなし(要約対象なし)」を提示する。AI 非対応・生成失敗は `Err` にせず degrade(NFR-V045-003)。
  - 生成テキストには生成ラベル(`mdi-creation`)を付与する。

### FR-V045-005: 1日1回バックグラウンド生成 + 手動再生成導線

- **優先度**: 必須
- **説明**: ①②の集約レポートはバックグラウンドで生成し、UI を開いた時点で準備済みにする。手動で即時再生成もできる。
- **受け入れ基準**:
  - スケジューラ(または起動時 setup)が「前回生成からの経過時間(横断サマリ ≈ 20時間以上)」「週/月のロールオーバ(`period_key` 未生成)」を判定し、AI 機能 ON のときだけバックグラウンドで生成する。`sync`・UI をブロックしない独立タスク(v0.4 の `sync_corpus_and_embeddings` と同様、本体を止めない)。
  - 生成は `job_queue` を介さずコマンド/スケジューラタスクから直接 `create_backend` → `infer` を呼ぶ(sidecar 直列管理タスクで analyze/embed と直列化)。
  - `/reports` 画面に**「再生成」**導線を各レポート種別に設け、手動で即時生成できる(`generate_reports(workspace_id, report_type, lang)` 等のコマンド)。生成中はスピナー・前回生成時刻を表示する。
  - AI 機能 OFF・可用性なし・キュー空のときは生成せずアイドル(CPU 非消費・NFR-V045-002)。

### FR-V045-006: レポートの保存・履歴閲覧

- **優先度**: 必須
- **説明**: 生成済みレポートを保存し、週次/月次は過去分を閲覧できる。
- **受け入れ基準**:
  - `report_summaries` テーブルを新設する。PK は `(workspace_id, report_type, period_key, lang)`。カラム: `stats_json`(決定的集計の保存)・`headline`・`narrative`・`generated_at`。
  - 横断サマリ(`cross_summary`)は `period_key='latest'` で最新のみ保持(上書き)。週次/月次は期間キーで履歴保持。
  - `/reports` 画面で最新を即表示し、週次/月次は期間セレクタで過去を切り替えられる。
  - ワークスペース削除・無効化・プロジェクト選択解除時に対応する `report_summaries` 行を孤児掃除する(v0.4 の削除系メソッドに準拠)。
  - **エクスポート(Markdown/ファイル書き出し・外部共有)はスコープ外**(将来)。

## 非機能要件

### NFR-V045-001: プライバシー(ローカル完結)

- 集約・要約生成はすべてオンデバイスで完結する。課題データ・コメント・集計結果を外部(クラウド API 等)へ送信しない。v0.3/v0.4 の差別化(ローカル完結・外部送信なし)を維持する。

### NFR-V045-002: リソース消費・性能

- 数値集計は SQL で行い、LLM へ渡す context は注目上位 N 件・`< CONTEXT_MAX_CHARS`(≈3000字)に抑える(3B 級のコンテキスト上限対策)。
- レポート生成はバックグラウンド・低頻度(1日1回 + 期間ロールオーバ)。AI 機能 OFF・可用性なし・生成不要(履歴最新が新しい/キャッシュ一致)のときは sidecar を起こさずアイドル。`sync`・UI をブロックしない。
- ③背景要約は `source_hash` キャッシュにより、コメント不変なら再生成しない。

### NFR-V045-003: 堅牢性(degrade)

- AI 非対応・生成失敗・対象データなし(`ai_results` ゼロ/コメントなし)でも、アプリ本体・既存機能・v0.3/v0.4 機能は継続動作する。レポート/要約のみ degrade し、理由(AI 未有効・データ未生成・コメントなし等)を提示する。
- 数値統計テーブルは LLM に依存せず常に表示できる(narrative のみ degrade)。

### NFR-V045-004: 互換性・配布

- Swift sidecar を改修しないため、配布物・ビルド手順は v0.4 から不変(再ビルド・再 codesign・再配布なし)。
- 既存テーブルは非破壊で拡張する(v0.4 の `IF NOT EXISTS` / `ALTER` エラー無視のインクリメンタルマイグレーション方式)。

## スコープ外(やらないこと)

- **レポートのエクスポート/外部共有**(Markdown・ファイル書き出し・Slack 連携等)→ 将来。v0.4.5 は画面内閲覧のみ。
- **日英同時生成(海外メンバー向け英語版の併存)**→ 将来。v0.4.5 は UI 言語追従のみ。
- **sidecar への自由文 `summarize` 経路の新設**→ 将来(レポート品質向上が必要になれば。`summarize_solutions` も同時移行可)。v0.4.5 は既存 `analyze` 流用。
- **map-reduce による全課題再要約**→ 不採用(既存 `ai_results` + SQL 集約の再利用で代替)。
- **任意期間/カスタム集計のレポートビルダー**(任意の開始終了日・任意ディメンション)→ 将来。v0.4.5 は横断(最新)・週次・月次の固定3種。
- **別ワークスペース(別 Backlog スペース)横断の混在サマリ**→ やらない(横断はワークスペース内のプロジェクト横断に限定)。
- **MLX(Qwen3)バックエンド** → さらに後続リリース(下記参照)。
- **課金・ライセンス・Free/Pro ゲート** → v0.5。

## 未解決事項(実装中に判断)

- **横断サマリの注目上位 N・重み付け**: 期限超過/高リスク/停滞/自分担当のどれをどう重み付けして上位を選ぶか。暫定: 期限超過日数と `risk_level` を主、停滞を従。N は context 上限に収まる範囲(目安 5〜10件)。
- **停滞しきい値・横断サマリの対象範囲**: 「停滞」と判定する未更新日数(目安 14日)。1ワークスペースに有効プロジェクトが多い場合の context 収まり(プロジェクト数が多いと統計だけで嵩む → narrative 対象を上位プロジェクトに絞る等)。
- **週/月のロールオーバ生成の起点**: 週境界(ISO 週=月曜開始)・月境界の判定と、初回起動が期間途中だった場合の扱い(部分期間レポートの可否)。日付は SQL(julianday / strftime)で決定的に算出し LLM に依存しない。
- **`analyze` 流用時の narrative 整形**: 構造化スキーマ(`summary`/`suggestion`)に複数行 narrative を載せる際の体裁(`suggestion` を本文・`summary` を見出しにマップ)。1行に収まらない場合の改行・箇条書き表現の安定性は実機で確認。品質が不足する場合は将来の自由文 `summarize` 経路へ移行。
- **`report_summaries.stats_json` のスキーマ**: プロジェクト別カウントの JSON 形状(UI 表示・将来の差分比較を見据える)。Zod/serde 両側の型定義箇所。
- **③背景要約の context 上限**: コメントが多い課題の切り詰め方針(`get_comments_text` の `max_chars`。決定事項が新しいコメントに偏るなら末尾優先か、時系列順か)。
- **生成タイミングの設定可否**: 1日1回の自動生成を設定で ON/OFF できるようにするか(既定 ON。AI 機能トグルに従属で足りるか)。
