# ProjectLens レポートの深い分析(優先対応リスト + 名指しブロッカー分析) 要件定義書(v0.4.6)

> **リリース**: v0.4.6(横断サマリの「優先対応リスト」+ sidecar 自由文生成による深い narrative)
> **ステータス**: 確定(2026-06-14、`/refine-requirements v0.4.6` の壁打ちにより確定)
> **前提**: v0.4.5(レポート/サマリー画面・横断サマリ/週次・月次・背景要約・`select_report_highlights` による注目上位 N 件のランキング)の完了・main マージ
> 起点: v0.4.5 の実機確認で「横断サマリの narrative が**通り一遍**で、どれから優先すべきか・何がブロッカーか・誰に確認すべきかが分からない」というユーザーフィードバック。

## 背景と目的

v0.4.5 の横断サマリは (a) プロジェクト別の**件数**しか UI に出さず、(b) narrative は単票分析用の `analyze` プロンプト流用 + 各課題の汎用1行要約(`ai_summary`)を入力にするため、出力が「優先順位をつけて対応・PM と確認」のような**抽象的な一般論**に流れる。一方で内部では注目上位 N 件を重み付けスコアで**ランク付け済み**(`select_report_highlights` / `report_highlight_score`)なのに、それを UI に出していない。

v0.4.6 は「**レポートを見ただけで ①どの課題から着手すべきか ②何がブロッカーか ③誰に確認すべきか が分かる**」を成功基準に、次の二段構えで深さを出す:

1. **決定的な「優先対応リスト」**: ランク済み個票を、課題キー・タイトル・**優先理由**(期限超過/内容リスク/停滞/担当状況)・**担当=連絡先**つきで可視化する。AI 品質に依存せず堅牢。これが最大の効き。
2. **sidecar 自由文生成による深い narrative**: 単票分析の `analyze` 流用をやめ、**自由記述の生成経路を sidecar に新設**。優先対応リストを入力に、**課題キーを名指し**で「今週の最優先と理由・ブロッカー・誰に確認」を言語化する。

### 設計上の確定事項(壁打ちより)

- **スコープは「優先対応リスト(決定的)+ sidecar 自由文要約」の両輪**。決定的な事実(どれ・なぜ・誰に)は SQL/ランクで確実に出し、文章での深い統合は新 sidecar 経路に任せる(v0.3/v0.4 の「数値は決定的・文章は LLM」を踏襲・強化)。
- **優先/ブロッカーのシグナルは4種すべて**(壁打ちで選択): ①期限超過(遅延日数)②内容リスク(`ai_results` の high/medium、FR-V04-006 の遅延合成込み)③停滞(一定日数 更新なし)④担当状況(未割当=要アサイン/PM 確認・担当あり=連絡先)。各課題に「なぜ優先か」として表示し、ランクにも効かせる。
- **表示は「横断クロスプロジェクト上位 N」+「プロジェクト別グループ」**(壁打ちで選択)。横断サマリ画面に出す。週次/月次は「期間の振り返り」性格のため優先リストは出さない(スコープ外)。
- **既存 `analyze` 流用 narrative は、レポートに限り新 sidecar 経路へ置換**。`summarize_solutions`(解決策要約)・`get_background_summary`(背景要約)は当面 `analyze` 流用のまま(移行は将来・任意)。

## アーキテクチャ方針(確定)

```
[優先対応リスト — 決定的]
  get_report_highlight_inputs を拡張(issue_key/ai_summary に加え title/assignee/status を返す)
      ↓
  report_highlight_score でランク(期限超過 + 内容リスク + 停滞、+ 担当状況の理由付与)
      ↓
  横断: 上位 N 件 / プロジェクト別: 各プロジェクト上位 K 件
      ↓
  Vue UI(優先対応リスト: 課題キー・タイトル・優先理由チップ・担当=連絡先・クリックで起票/背景要約)

[深い narrative — sidecar 自由文生成]
  優先対応リスト(課題キー + タイトル + 理由 + 担当) + プロジェクト集計 を compact context 化
      ↓
  sidecar の新経路 {type:"summarize", instruction, context, lang} → 自由記述生成(@Generable 非拘束)
      ↓
  report_summaries.narrative(課題キーを名指し・今週の最優先と理由・ブロッカー・誰に確認)
      ↓
  Vue UI(ReportNarrative: 生成ラベル付き。AI 非対応・生成失敗は優先対応リストのみで degrade)
```

## 機能要件

### FR-V046-001: 決定的「優先対応リスト」の算出と表示

- **優先度**: 必須
- **説明**: ランク済みの個別課題を、優先理由・担当つきで可視化する。横断サマリの中核を「件数」から「どの課題から」へ引き上げる。
- **受け入れ基準**:
  - `get_report_highlight_inputs`(または新メソッド)を拡張し、`issue_key` / `ai_summary` に加え **`title`(課題タイトル=`issues.summary`)・`assignee`・`status`** を返す(いずれも `issues` テーブルに既存)。
  - ランクは既存 `report_highlight_score`(期限超過日数 + 内容リスク + 停滞)を踏襲しつつ、**担当状況**を理由表示に加える。スコアの重み・上位件数(横断 N、プロジェクト別 K)は定数で一元定義(暫定: 横断 N=8〜10、プロジェクト別 K=3〜5)。
  - UI(横断サマリ画面)に「**優先対応リスト**」を表示する。各行: プロジェクトキー・課題キー・タイトル・**優先理由チップ**(下記 FR-V046-002)・**担当=連絡先**(未割当は「要アサイン」)。類似度降順ならぬ**優先スコア降順**。
  - 各行クリックで (a) Backlog で起票を開く、(b) その課題の「**背景・経緯を要約**」(v0.4.5 実装済み)へ導線。完了課題(コーパス)は対象外、未完了(通常 sync)課題のみ。
  - 統計テーブル(件数)は維持しつつ、優先対応リストを主役に据える。データ無し・未生成時は degrade 文言。

### FR-V046-002: 優先/ブロッカーシグナルの決定的算出と理由表示

- **優先度**: 必須
- **説明**: 各課題の「なぜ優先か/何がブロッカーか/誰に確認すべきか」を、決定的なシグナルとして算出・表示する。
- **受け入れ基準**:
  - 次の4シグナルを各課題に付与する(すべて SQL/決定的・ローカル日基準):
    - **期限超過(遅延日数)**: `delay_days > 0`。例「469日超過」。
    - **内容リスク(AI)**: `ai_results.risk_level` の high/medium(FR-V04-006 の遅延合成済み)。例「高リスク」。
    - **停滞(更新なし)**: `updated_at` がしきい値(暫定14日)以上前。例「停滞28日」。
    - **担当状況**: 未割当 → 「**要アサイン(担当決め)**」、担当あり → 「**{担当者名} に確認**」。これが「誰に確認すべきか」を直接満たす。
  - 該当するシグナルのみをチップ等で簡潔に表示する(0個はリスト入りしない=ランク下位)。色は重大度に準じる(期限超過/高リスク=赤系、停滞=橙、未割当=情報系)。
  - しきい値・重み付けは定数に集約し、`recompute`/SQL に二重定義しない(FR-V04-006 の `schedule_risk` と同方針)。
  - i18n(ja/en)で理由ラベルを提供する。

### FR-V046-003: sidecar 自由文生成経路の新設

- **優先度**: 必須
- **説明**: レポート narrative を「単票分析の `analyze` 流用」から「自由記述生成」へ切り替えるため、Swift sidecar に新しい生成経路を追加する。
- **受け入れ基準**:
  - sidecar に新リクエスト `{type:"summarize", instruction, context, lang}` と新レスポンス `{type:"summarize", text}` を追加する(`@Generable` 拘束のない自由記述生成。FoundationModels の text generation を使う)。`availability`/`analyze`/`embed`/`shutdown` と同じ管理タスク・直列処理に乗せる。
  - Rust 側に `SidecarRequest::Summarize { instruction, context, lang }` / 対応レスポンス・`ManagerCommand`・`ExpectedResponse` を追加し、`FoundationModelsBackend` に `summarize(&self, instruction, context, lang) -> Result<String>` を実装する(`infer`/`embed` と同じ突合・再起動・一時停止の枠組み)。
  - instructions は guided generation スキーマと合算でコンテキストを圧迫しないよう簡潔に保つ(v0.3 の教訓)。context は優先対応リスト中心に上限文字数で切り詰める(NFR-V046-002)。
  - **ビルド統合**: sidecar 改修に伴う `swift build` 再ビルド・codesign・externalBin 再配置は `build.sh` の既存フローで吸収する。プロトコル契約は `src-tauri/sidecar/README.md` と Rust 側で一致させる。
  - 失敗・タイムアウト時は `Err` を返し、呼び出し側(narrative 生成)が degrade する(NFR-V046-003)。

### FR-V046-004: 優先対応リストを入力にした深い narrative 生成

- **優先度**: 必須
- **説明**: 新 sidecar 経路で、課題キーを名指しした「今週の最優先・ブロッカー・誰に確認」の narrative を生成する。
- **受け入れ基準**:
  - context = 優先対応リスト(**課題キー + タイトル + 優先理由 + 担当**)+ プロジェクト別集計(件数)を compact に組み立てる(`ai_summary` の汎用1行ではなく、決定的シグナルとタイトルを与える)。
  - instruction は「列挙された課題を優先度順に評価し、**最優先の数件を課題キーで名指し**し、それぞれの**ブロッカー(なぜ進んでいないか)**と**誰に確認・何をすべきか**を簡潔に述べる」よう指示する。一般論の列挙を避ける方向に明示的に寄せる。
  - 出力言語は UI 言語追従(ja/en)。`report_summaries.narrative` に保存し、横断サマリの ReportNarrative に生成ラベル付きで表示する。
  - レポート narrative の生成経路は、**レポートに限り `analyze` 流用から本経路へ置換**する(`generate_report_narrative` を改修)。`summarize_solutions`・`get_background_summary` は当面 `analyze` 流用のまま据え置き(移行は将来・任意。本リリースでは変更しない)。
  - 数値・課題キー・担当などの**事実は決定的データを真とする**(LLM 出力の数値は信用しない。narrative はあくまで統合・言語化)。

### FR-V046-005: degrade と非阻害

- **優先度**: 必須
- **説明**: AI 非対応・生成失敗でも、優先対応リスト(決定的)は常に表示できる。
- **受け入れ基準**:
  - AI 非対応・`summarize` 失敗・対象なしのとき、narrative は空へ degrade し、**優先対応リスト(決定的)と統計テーブルは表示を維持**する(NFR-V046-003)。
  - 生成はバックグラウンド(v0.4.5 の `generate_due_reports`)と手動再生成の両経路で本経路を通る。本体 sync・UI をブロックしない。

## 非機能要件

### NFR-V046-001: プライバシー(ローカル完結)

- 優先対応リストの算出・narrative 生成はすべてオンデバイスで完結する。課題データを外部へ送信しない(v0.3〜v0.4.5 を維持)。

### NFR-V046-002: 性能・コンテキスト上限

- 数値・ランク・シグナルは SQL/Rust で決定的に算出する。`summarize` へ渡す context は優先対応リスト中心に上限文字数(目安 `SUMMARIZE_CONTEXT_MAX_CHARS` 級・≈3000字)で切り詰め、3B 級のコンテキスト上限に収める。
- レポート生成は低頻度(1日1回 + 期間ロールオーバ + 手動)。アイドル時は sidecar を起こさない(v0.4.5 の due 判定先行を維持)。

### NFR-V046-003: 堅牢性(degrade)

- AI 非対応・生成失敗でもアプリ本体・既存機能は継続。レポートは「優先対応リスト + 統計のみ」へ degrade し、narrative の不在理由を提示する。

### NFR-V046-004: 互換性・配布

- sidecar 改修(新 `summarize` 経路)に伴い再ビルド・codesign・externalBin 再配置が必要(v0.4.5 までは不要だった点からの差分)。`build.sh` の既存フローで吸収し、sidecar ビルド失敗時は AI 機能なしで本体ビルドを継続(既存フォールバック)。
- DB スキーマ拡張は非破壊(`IF NOT EXISTS` / `ALTER` エラー無視)。`report_summaries` は v0.4.5 の構造を流用(narrative の出所が変わるだけ)。

## スコープ外(やらないこと)

- **週次/月次レポートへの優先対応リスト展開** → 将来(期間レポートは振り返り性格のため。v0.4.6 は横断サマリのみ)。
- **`summarize_solutions`・`get_background_summary` の自由文経路への移行** → 将来・任意(本リリースは `analyze` 流用のまま据え置き)。
- **ブロッカーのカスタムルール定義 UI**(ユーザーが任意条件を設定)→ 将来。v0.4.6 は固定4シグナル。
- **依存関係/ブロック状態の自動抽出**(Backlog の課題リンク・「ブロック」状態の取得)→ 将来(本リリースのブロッカーは期限超過/リスク/停滞/担当状況の決定的シグナルに限定)。
- **PM の自動特定** → やらない(Backlog にロール情報がないため。未割当は「要アサイン(担当決め)」として PM/リーダー確認を促す表示に留める)。
- **MLX(Qwen3)バックエンド・課金** → 後続/v0.5。

## 未解決事項(実装中に判断)

- **件数の既定値**: 横断 N(目安 8〜10)・プロジェクト別 K(目安 3〜5)。画面の見やすさと context 上限の両立で実データ調整。
- **担当状況の重み**: 未割当をランクスコアに加点するか(「放置 + 未割当」を上位に出すか)、理由表示のみに留めるか。
- **`summarize` の出力整形**: 自由記述の改行・箇条書き・課題キー名指しの安定性は FoundationModels の出力依存。instruction の作り込みと実機検証で調整。長すぎる出力の上限・トリム。
- **プロジェクト別グループの UI 形状**: 横断上位 N とプロジェクト別グループの併存レイアウト(タブ/アコーディオン/2カラム)。情報過多にならない見せ方。
- **`analyze` 流用の撤去範囲**: レポートのみ置換で確定だが、`generate_report_narrative` の旧経路コードを残すか削除するか(将来の比較用に残す選択肢)。
- **既存 `report_summaries` 行の扱い**: v0.4.5 で生成済みの narrative(analyze 流用)を、初回起動時に再生成するか自然更新(次の due)に任せるか。
